## Что было сделано в 1 семестре

- Сравнивались два подхода:
  - прямой `audio -> tablature` подход на стороне Вали, связанный с `TabCNN`
  - пайплайн `audio -> MIDI -> tablature` на стороне Liss
- В пайплайне Liss:
  - для `audio -> MIDI` использовался `BasicPitch`
  - для `MIDI -> tablature` использовался трансформер, который переводил питч в набор параметров `string-fret`
  - дополнительно вводилась метрика `playability`, чтобы предсказания были удобны для реальной игры
- Основная проблема была в качестве шага `audio -> MIDI`
- Постпроцессинг частично помогал, но мог удалять короткие ноты, которые иногда были важны для мелодии
- Шаг `MIDI -> tablature` работал заметно лучше, примерно на `90%` точности
- По итогам 1 семестра команда решила, что по качеству прямой `audio -> tablature` подход выглядел слабее, чем пайплайн `audio -> MIDI -> tablature`

## Текущее понимание после проверки

- Старый пайплайн Liss подтверждает, что узкое место находится в `audio -> MIDI`, а не в `MIDI -> tablature`
- В `tab-cnn` есть частичная миграция на Python 3 и `tensorflow.keras`, но это еще не готовый современный и удобный экспериментальный пайплайн
- В `SynthTab` есть более подходящая база для дальнейших экспериментов:
  - PyTorch-обучение
  - датасет-обертки
  - поддержка GPU в коде
  - утилиты `JAMS <-> MIDI`
- Готовой реализации `LoRA` в проверенных локальных репозиториях нет
- Для проверки идеи `LSTM` или рекуррентного варианта логичнее опираться на recurrent-ветку из стека `SynthTab`
