# ТЗ для Люмы: следующий этап проекта Guitar Tabs

Проект посвящён распознаванию гитарных табулатур по аудиосигналу.

Основной репозиторий:

`D:\PP-VKL-GuitarTabs`

В проекте уже соединены подходы TabCNN и SynthTab: код SynthTab адаптирован под обучение TabCNN-подобной модели на данных SynthTab Dev, есть train/eval/inspect режимы, CUDA-поддержка, конфиги и сохранённые результаты экспериментов.

Перед началом новой работы обязательно повторно изучить проектные логи и handoff-файлы:

- `D:\PP-VKL-GuitarTabs\PROJECT_LOG.md`
- `D:\PP-VKL-GuitarTabs\workspace\HANDOFF_NEXT_CHAT.md`
- `D:\PP-VKL-GuitarTabs\README.md`

Эти файлы являются источником правды по уже выполненной работе, текущим метрикам, структуре проекта и предыдущим решениям.

## 2. Что уже сделано

Уже реализовано:

- код обучения на SynthTab Dev;

Эти значения нужно использовать как baseline для сравнения дальнейших запусков.

## 3. Важное уточнение про SynthTab

Текущий `SynthTab_Dev.zip` — это не полный датасет.

Это development subset, предназначенный для отладки пайплайна.

Dev-набор:

https://rochester.app.box.com/v/SynthTab-Dev

Полный датасет SynthTab:

https://rochester.app.box.com/v/SynthTab-Full

Официальный full SynthTab занимает почти `2 TB`, содержит примерно `6 700` часов аудио, `15 211` треков и `23` тембра.

Поэтому нельзя строить план так, будто весь датасет можно просто скачать и распаковать локально.

## 4. Ограничение по месту на диске

На текущих устройствах нет свободных 2 TB:

- на ноутбуке около `0.5 TB`;
- на компьютере около `1 TB`, но примерно половина уже занята;
- полный SynthTab почти `2 TB`.

Нужно продумать стратегию обучения на максимально возможном объёме данных при нехватке места.

Запрещённый подход:

- скачать весь SynthTab целиком;
- распаковать всё сразу;
- только после этого начинать работу.

Рекомендуемый подход:

- скачать одну часть датасета;
- распаковать или подключить только её;
- обучить модель на этой части;
- сохранить полный checkpoint;
- удалить или заменить текущую часть;
- скачать следующую часть;
- продолжить обучение через resume.

## 5. Стратегия работы с full SynthTab

Нужно проверить структуру full SynthTab по ссылке:

https://rochester.app.box.com/v/SynthTab-Full

Ожидаемая структура включает:

- `all_jams_midi_V2_60000_tracks.zip`;
- acoustic-части;
- electric clean;
- electric distortion;
- electric muted;
- отдельные zip-архивы по группам/тембрам.

Желательная стратегия:

1. Постоянно хранить локально JAMS/MIDI, так как этот архив значительно меньше аудио.
2. Аудио хранить и обрабатывать частями.
3. Обучать модель последовательно по чанкам.
4. После каждого чанка сохранять checkpoint и метрики.
5. При необходимости удалять старые чанки для освобождения места.
6. Продолжать обучение на следующем чанке.

Если обучение на всех данных физически невозможно, собрать сбалансированный subset:

- acoustic;
- electric clean;
- electric distortion;
- electric muted;
- разные timbres;
- разные уровни плотности нот;
- разные треки, а не много фрагментов одного типа.

## 6. Сделать полноценный resume обучения

Сейчас в проекте есть сохранение чекпоинтов, но полноценное продолжение обучения нужно доработать.

Нужно, чтобы обучение можно было остановить и продолжить позже без потери состояния.

В checkpoint нужно сохранять:

- model state;
- optimizer state;
- scheduler state;
- epoch;
- global step / `model.iter`;
- random state Python;
- random state NumPy;
- random state Torch;
- CUDA random state, если используется GPU;
- snapshot конфига обучения;
- информацию о текущем chunk/subset датасета.

В конфиг добавить параметр вида:

```json
{
  "resume_from": "path/to/checkpoint.pt"
}
```

После resume обучение должно продолжаться с сохранённого шага, а не начинаться заново.

Обязательно проверить сценарий:

1. запустить короткое обучение;
2. дождаться checkpoint;
3. остановить процесс;
4. запустить resume;
5. убедиться, что epoch/global step продолжаются, а не сбрасываются.

## 7. Настроить сбалансированные батчи

Сейчас в коде есть обычное перемешивание через DataLoader:

```python
DataLoader(..., shuffle=True)
```

Это лучше, чем отсутствие shuffle, но недостаточно для большого и неоднородного датасета.

Нужно добавить balanced sampling или balanced batch sampler.

Цели:

- не давать модели слишком много silent-фреймов подряд;
- уменьшить риск collapse-to-silence;
- равномернее использовать разные треки;
- смешивать разные timbres;
- не допускать перекоса в один тип гитары или один тип аудио;
- сохранить воспроизводимость через seed.

Рекомендуемая стратегия балансировки:

- silent / non-silent ratio;
- acoustic / electric группы;
- clean / distortion / muted;
- timbre group;
- track-level diversity.

Обычный shuffle оставить как fallback.

Class weighting с `silence_weight = 0.100` оставить включённым как текущую лучшую настройку.

## 8. Порядок запуска обучения

Порядок работы:

1. Повторно изучить `PROJECT_LOG.md`, `HANDOFF_NEXT_CHAT.md` и `README.md`.
2. Проверить, что текущий SynthTab Dev запускается.
3. Запустить smoke-test на SynthTab Dev.
4. Подключить маленькую часть full SynthTab.
5. Запустить smoke-test на части full SynthTab.
6. Проверить сохранение checkpoint.
7. Проверить resume.
8. Проверить balanced batches.
9. Запустить обучение на первой крупной части full SynthTab.
10. Сохранять метрики после каждого этапа.
11. Продолжать обучение по следующим частям датасета.

Для full-run снять ограничения вида:

```json
{
  "limit_train_tracks": null,
  "limit_val_tracks": null
}
```

или заменить их на chunk-based логику.

## 9. Метрики и логи

После каждого запуска сохранять:

- loss;
- multi-pitch precision;
- multi-pitch recall;
- multi-pitch F1;
- tablature precision;
- tablature recall;
- tablature F1;
- non-silent accuracy;
- pred_silence_ratio;
- collapse_to_silence;
- путь к checkpoint;
- параметры обучения;
- какой subset/chunk/timbre использовался;
- дата и длительность запуска;
- была ли это новая тренировка или resume.

Важно: результаты нужно сохранять так, чтобы потом можно было объяснить преподавателям:

- какие данные использовались;
- почему использовался не весь датасет сразу;
- какие параметры подбирались;
- какие метрики улучшились;
- какие ограничения остались.

## 10. Дальнейшие планы после обучения

После обучения на большем объёме данных нужно сделать сравнение с другими сервисами или подходами.

Сравнение проводить на одинаковых аудиофрагментах.

Фиксировать:

- качество распознавания нот;
- корректность струн;
- корректность ладов;
- пропуски нот;
- лишние ноты;
- читаемость итоговой табулатуры;
- пригодность результата для музыканта.

Этот блок понадобится для отчёта и презентации.

## 11. Финальная цель проекта

После подбора параметров и обучения на большем объёме SynthTab нужно попробовать дальнейшую адаптацию модели на собственных данных.

Собственные данные будут состоять из:

- табулатур, написанных вручную;
- аудио игры гитариста, который играет по этим табам.

Идея дальнейшего этапа:

- использовать SynthTab/pretrained model как базу;
- затем дообучить модель на собственных данных;
- возможно применить LoRA или другой метод лёгкой адаптации.

## 12. Acceptance Criteria

Работа считается выполненной, если:

- повторно изучены проектные логи;
- подтверждено, что SynthTab Dev — не полный датасет;
- зафиксирована стратегия работы с full SynthTab при нехватке места;
- обучение можно остановить и продолжить из checkpoint;
- реализованы или спроектированы сбалансированные батчи;
- проведён хотя бы один успешный запуск на части full SynthTab;
- сохранены метрики и логи;
- понятно, какие данные использовались на каждом этапе;
- есть план сравнения с другими сервисами;
- проект готов к дальнейшему дообучению на собственных данных.
